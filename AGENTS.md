## Stack

TanStack Start + TanStack Router (SSR) + React 19 + Tailwind CSS v4 + shadcn/ui (new-york style, slate base, lucide icons) + Supabase + Recharts + date-fns. Built with Vite 8, deployed to Cloudflare Workers via Nitro (`cloudflare-module` preset).

## Architecture

- **Entrypoints:** `src/server.ts` (production fetch handler for CF Workers), `src/start.ts` (TanStack Start config + middleware)
- **Routes:** Flat `.tsx` files under `src/routes/` — `__root.tsx`, `index.tsx`, `tributarias.tsx`, `parafiscales.tsx`, `libros.tsx`
- **`src/routeTree.gen.ts`** auto-generated by TanStack Router — do not edit, but committed to repo
- **App entry:** `src/components/app-shell.tsx` (sidebar + header layout)
- **Data hook:** `src/hooks/use-dashboard-data.ts` — queries Supabase tables `iva_spe` / `iva_spo` via `useSuspenseQuery`
- **Mock data fallback:** `src/lib/mock-data.ts` — used for `conceptosPorTab` filter chips; `parafiscalesRows` and `librosRows` are currently hardcoded empty arrays

## Conventions & quirks

- **`@/` alias** maps to `src/` (configured in tsconfig.json + vite.config.ts)
- **`noUnusedLocals` / `noUnusedParameters` are off** in tsconfig; **`@typescript-eslint/no-unused-vars` is off** in ESLint
- **`verbatimModuleSyntax: false`** — mixing `import` and `import type` is fine
- **`routeTree.gen.ts`** is in `.prettierignore` but committed to git
- CSS uses **oklch** color format with Tailwind v4 `@theme inline` + Lightning CSS (not PostCSS)
- Two Supabase clients: `client.ts` (browser, `VITE_SUPABASE_*` env vars via `import.meta.env`) and `client.server.ts` (server service-role, `SUPABASE_*` via `process.env`, lazy Proxy init)
- **Missing DB tables** — the code queries `iva_spe` and `iva_spo` but only `clients`, `responsibles`, `obligations`, `contacto` exist in migrations at `supabase/migrations/`
- Server-only code uses `.server.ts` naming convention (TanStack Start); importing `server-only` is restricted by ESLint

## graphify

This project has a knowledge graph at graphify-out/ with god nodes, community structure, and cross-file relationships.

When the user types `/graphify`, use the installed graphify skill or instructions before doing anything else.

Rules:

- For codebase questions, first run `graphify query "<question>"` when graphify-out/graph.json exists. Use `graphify path "<A>" "<B>"` for relationships and `graphify explain "<concept>"` for focused concepts. These return a scoped subgraph, usually much smaller than GRAPH_REPORT.md or raw grep output.
- Dirty graphify-out/ files are expected after hooks or incremental updates; dirty graph files are not a reason to skip graphify. Only skip graphify if the task is about stale or incorrect graph output, or the user explicitly says not to use it.
- If graphify-out/wiki/index.md exists, use it for broad navigation instead of raw source browsing.
- Read graphify-out/GRAPH_REPORT.md only for broad architecture review or when query/path/explain do not surface enough context.
- After modifying code, run `graphify update .` to keep the graph current (AST-only, no API cost).
